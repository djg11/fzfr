# Agent Contributor Guidelines for fzfr

This document provides instructions for AI agents contributing to `fzfr`.
Your role is that of a **Senior Implementer** -- you write, test, and refactor
code according to the architectural and stylistic rules defined here. The human
user remains the **Architect** who makes final design decisions.

These instructions are directive, not suggestions. Failure to follow them
produces broken builds and rejected contributions. Read this file completely
before making any changes.

---

## 1. Critical Project Mandates

### 1.1. Single-File Architecture is Paramount

The primary output of this project is a **single, self-contained Python
script** (`fzfr`, no extension). This monolithic design is the core
requirement for SSH remote search, which works by transferring this single
script to a remote host via stdin.

- MANDATE: All changes must preserve the integrity of the single-file build
  process (`scripts/build_single_file.py`).
- MANDATE: Any change that breaks the ability of the built script to bootstrap
  itself on a remote machine is a critical failure.
- MANDATE: Never edit the top-level `fzfr` file directly. It is a build
  artifact, gitignored, and regenerated by `make build`.

### 1.2. Source of Truth is `src/fzfr/`

All development MUST occur in the Python modules within `src/fzfr/`.

### 1.3. fzfr is a Wiring Layer

fzfr wires existing best-in-class CLI tools together. It does not reinvent
their features.

- MANDATE: Before adding any feature, verify it cannot be expressed as a
  user-defined custom action (`"cmd": "tool {path}"`). If it can, it does not
  belong in core.
- MANDATE: fzfr must degrade gracefully. New features must not hard-fail if an
  optional dependency (`bat`, `rga`, `tmux`, etc.) is absent. Always provide a
  fallback. See the Tier 1-5 tool stack in `TODO.md`.

## 2. Development Workflow

Every change follows this loop. Do not skip steps.

```
code
  -> refactor (simplify, extract helpers, reduce complexity)
  -> docstrings and comments (every function and module documented)
  -> format (line length, whitespace, import order)
  -> security check (quoting, path safety, semgrep)
  -> back to refactor if security check reveals structural issues
  -> make build (must pass, 0 errors)
  -> make test (all tests must pass)
  -> semgrep scan --config .semgrep/semgrep.yml --error . (0 findings)
  -> stash / commit / merge
```

Cycle through refactor -> security -> refactor at most once. If a second
pass reveals further structural problems, stop and raise the issue with the
Architect rather than iterating indefinitely.

### 2.1. Step by Step

1. **Modify source** in `src/fzfr/` only.
2. **Refactor:** simplify logic, extract helpers, reduce cyclomatic complexity.
   Target: no function over 10 branches or 80 lines without justification.
3. **Docstrings and comments:** every module and public function must have a
   docstring. Use `DESIGN:`, `SECURITY:`, `PERF:` keywords. ASCII only.
4. **Format:** 4-space indent, 88-char line limit, no trailing whitespace,
   one import per line, no ambiguous variable names (`l`, `O`, `I`).
5. **Lint and format:** run `ruff check --fix src/ tests/` then
   `ruff format src/ tests/`. Fix any remaining warnings manually.
6. **Security check:** review all subprocess calls, path operations, and SSH
   command construction against the rules in section 5. Run semgrep.
7. **If security check reveals structural issues** (e.g. a function mixes
   command construction with execution making safe quoting hard): go back to
   step 2 and refactor first, then re-check.
8. **Build:** `make build` -- regenerates `fzfr` and runs the test suite.
9. **Verify:** `semgrep scan --config .semgrep/semgrep.yml --error .`

Never commit with a failing build, failing tests, or semgrep findings.

## 3. Build System Internals

Understanding the build is essential for avoiding subtle bugs.

### 3.1. How `build_single_file.py` Works

The script concatenates all modules in `MODULE_ORDER` into a single Python
file:

1. Strips each module's docstring and all import statements.
2. Collects stdlib/third-party imports via `ast.walk` (not line scanning --
   line scanning would match lines inside docstrings).
3. Deduplicates and sorts collected imports into one block at the top.
4. Appends each module body under a `# === module.py ===` header.
5. Runs `ast.parse` on the result -- fails hard if there is a syntax error.

### 3.2. Rules for Writing Build-Safe Code

**Imports must be at module top level.**
All `import` and `from X import Y` statements must appear at the top of the
module file, outside any function or class. The build script strips them
before concatenation and re-emits them once at the top of the built file.
A late import inside a function body will be stripped and leave the name
undefined in the flat file.

Exception: intra-package imports (`from .X import Y`) inside `try/except
ImportError` blocks are handled specially for circular-import resolution.
See `backends.py` for the established pattern.

**Do not start docstring or comment lines with `from ` or `import `.**
The build script uses `ast.walk` to collect imports, so prose sentences that
happen to start with "from" are safe -- but keep prose unambiguous anyway.
Write "sourced from X" rather than "from X, the value is sourced".

**No multi-line string literals containing triple-quote sequences.**
The build script does not track string context when stripping. Avoid
embedding `"""` inside other `"""` strings.

**The `typing` import pattern.**
`from typing import NamedTuple`, `Protocol`, `Callable` etc. must be at
module top level. If inside a function, the build strips them and
`class Foo(NamedTuple):` fails with a `NameError` in the flat file.

### 3.3. Module Order

Modules are concatenated in this order (defined in `MODULE_ORDER`):

```
_script, utils, workbase, config, tty, ssh, state, cache,
archive, backends, preview, internal, dispatch,
open, copy, remote, search
```

A module may reference symbols from any earlier module in this list without
importing them -- they are already in scope in the flat file. Intra-package
imports (`from .config import CONFIG`) are stripped and exist only to satisfy
the per-module linter and type-checker.

## 4. Code Style and Conventions

### 4.1. Formatting

- Indentation: 4 spaces. No tabs.
- Line length: 88 characters maximum.
- No trailing whitespace.
- Use `ruff format` for formatting. Run `ruff check --fix` for auto-fixable
  lint issues. See `pyproject.toml` for the full rule set.

### 4.2. Imports

- Standard library and third-party: absolute imports at module top level.
- Other fzfr modules: relative imports (`from .config import CONFIG`).
- Do not rely on import-time side effects; imports are deduplicated globally.
- Each import on its own line. No `import os, sys` on one line (Ruff E401).
- Avoid single-letter variable names `l`, `O`, `I` (Ruff E741).

### 4.3. Comments and Docstrings

- ASCII characters only. No em-dashes, unicode arrows, bullets, or box-drawing
  characters in comments or docstrings.
- Prose dashes: `--` not unicode em-dash. Arrows: `->` not unicode arrow.
- Section dividers in comments: `# -- Title --` or `# ===`.
- Bullet points in comments: `-` not `*` or unicode bullets.

Use these prefixed keywords for significant comments:

| Keyword    | Use for |
|------------|---------|
| `DESIGN:`  | Architectural choices, non-obvious patterns |
| `SECURITY:`| Input sanitization, shell command construction, path checks |
| `PERF:`    | Performance optimizations or trade-offs |
| `FIXME:`   | Temporary issues needing a follow-up |
| `TODO:`    | Planned future enhancements |

Example:
```python
# SECURITY: shlex.quote() all user-supplied paths before substitution.
# A crafted filename containing semicolons could inject shell commands
# at the second shell level (remote SSH execution).
safe_path = shlex.quote(path)
```

### 4.4. Docstrings

Every module and every public function must have a docstring.

Module docstrings:
```python
"""fzfr.module_name -- one-line summary.

Extended description: what the module owns, what it exports,
and any non-obvious design constraints.
"""
```

Function docstrings: one summary line, blank line, body if needed.
Include a `Usage:` line for all `cmd_*` entry points showing argv layout.

## 5. Security Constraints

These rules are enforced by `.semgrep/semgrep.yml`. Violations block CI.

### 5.1. Shell Injection Prevention

fzfr uses three quoting strategies deliberately -- use the correct one:

| Strategy | When to use |
|----------|-------------|
| List-form `["cmd", arg1, arg2]` | All local subprocess calls. The OS passes each element as a separate argv token -- no shell sees it. |
| `shlex.join(["cmd", arg])` | SSH remote commands and fzf `--bind` strings. SSH concatenates all args after the hostname into one string which the remote shell word-splits. |
| `_dquote(path)` | Editor commands travelling through two shell levels: `tmux new-window` -> `ssh` -> remote shell. Single-quoted strings break at the first level. |

- MANDATE: Never use `subprocess.run(cmd, shell=True)` for commands built
  from user input. The one legitimate use is the user-defined custom action
  command string in `internal.py:cmd_internal_exec`, annotated with
  `# nosemgrep: fzfr-subprocess-shell-true` and a justification comment.
- MANDATE: Never use `os.system()`.
- MANDATE: All fzfr placeholders (`{path}`, `{paths}`, `{dir}`, `{base}`,
  `{q}`) are `shlex.quote()`'d by `_substitute_placeholders` before any
  command string reaches the shell.

### 5.2. Path Safety

- All file operations on user-supplied paths must call
  `backend.is_safe_subpath(path)` before proceeding.
- State file paths arrive as argv elements from fzf callbacks and could be
  attacker-controlled. Always use `_load_state()` (which validates against
  `WORK_BASE`) rather than reading JSON directly with `Path.read_text()`.
- Session directories live under `WORK_BASE` (`/dev/shm/fzfr/` on Linux).
  `/dev/shm` is world-writable -- check for symlinks before and after `mkdir`
  using `_assert_not_symlink()`.

### 5.3. SSH Remote Commands

- All values interpolated into remote shell command strings must go through
  `shlex.quote()` or `shlex.join()`.
- `SCRIPT_HASH` is verified as hex-only (`re.fullmatch(r"[0-9a-f]{16}", ...)`)
  before interpolation in `_upload_remote_script`.

### 5.4. Exclude Patterns and Extensions

- Extension strings go through `_parse_extensions()`: alphanumeric-only.
- Exclude patterns go through `_validate_exclude_pattern()`: rejects shell
  operators. Do not bypass either function.

## 6. Architecture Reference

### 6.1. Module Responsibilities

| Module | Owns |
|--------|------|
| `_script.py` | VERSION, SELF path, SCRIPT_BYTES, bootstrap constants. No intra-package imports. |
| `utils.py` | _capture, _passthrough, _try_run, MIME detection, extension/exclude validation |
| `workbase.py` | WORK_BASE selection (/dev/shm or tmp), symlink safety checks |
| `config.py` | _CONFIG_DEFAULTS, load_config(), validation, CONFIG and AVAILABLE_TOOLS singletons |
| `tty.py` | _tty_prompt() -- /dev/tty prompt helper |
| `ssh.py` | _ssh_opts(), _ssh_opts_str() -- ControlMaster flag construction |
| `state.py` | _save_state(), _load_state(), _mutate_state() -- JSON session state |
| `cache.py` | _PreviewCache -- file-backed LRU preview output cache |
| `archive.py` | FileKind enum, classify(), _list_archive() |
| `backends.py` | SearchContext, LocalBackend, RemoteBackend, backend_from_state() |
| `preview.py` | cmd_preview (fzfr-preview) -- file preview rendering |
| `internal.py` | All _internal-* fzf callbacks, box renderer, which-key menu |
| `dispatch.py` | cmd_dispatch -- routes preview/reload to the session backend |
| `open.py` | cmd_open (fzfr-open) -- file/directory open logic |
| `copy.py` | cmd_copy (fzfr-copy) -- clipboard copy, remote path resolution |
| `remote.py` | cmd_remote_reload, cmd_remote_preview -- SSH remote search/preview |
| `search.py` | cmd_search (fzfr) -- main fzf UI, session lifecycle |
| `__init__.py` | COMMAND_MAP, main() entry point |

### 6.2. Session Lifecycle

```
cmd_search()
  -> create session_dir under WORK_BASE (mode 0o700)
  -> freeze script copy to session_dir/fzfr-frozen.py
  -> write initial state to session_dir/state.json
  -> Popen fzf, save fzf_pid to state
  -> fzf callbacks fire as subprocesses:
       _internal-dispatch   -> cmd_dispatch -> backend.preview / backend.reload
       _internal-toggle-*   -> _mutate_state
       _internal-prompt     -> _tty_prompt -> _mutate_state
       _internal-action-menu -> SIGSTOP fzf, which-key menu, SIGCONT fzf
       fzfr-open            -> cmd_open -> _open()
  -> on exit: _cleanup() removes session_dir and SSH socket
```

### 6.3. Custom Action System

Custom actions are user-defined shell commands bound to a two-level which-key
menu (leader key -> group key -> action key).

The leader bind fires `execute-silent`, starting `cmd_internal_action_menu`
as a subprocess. That subprocess immediately sends SIGSTOP to the fzf process
(PID stored in state), takes exclusive ownership of /dev/tty, draws the
which-key box, reads keypresses, then SIGCONTs fzf on exit.

Output modes:
- `"silent"` -- run silently, surface stderr on non-zero exit
- `"tmux"` -- new tmux window, SSH-wrapped for remote sessions
- `"overlay"` -- run while fzf frozen, show stdout in a terminal box,
  any key to dismiss, then SIGCONT

fzf-native output modes (`"fzf-preview"`, `"fzf-results"`, `"fzf-header"`)
are planned. See `TODO.md`.

### 6.4. fzf Placeholder Quoting

fzf expands `{}`, `{q}`, and `{+}` into shell-escaped tokens automatically.
These placeholders MUST be left unquoted in `--preview` and `--bind` strings.
Quoting them (e.g. `'{q}'`) causes double-escaping and breaks filenames with
spaces or special characters.

The custom action placeholders (`{path}`, `{paths}`, `{dir}`, `{base}`, `{q}`)
are substituted by `_substitute_placeholders()` in Python before reaching the
shell, so they MUST be `shlex.quote()`'d there.

## 7. Testing

### 7.1. Running Tests

```sh
make build                                        # build + test
make test                                         # test only, verbose
make lint                                         # ruff check + ruff format check
semgrep scan --config .semgrep/semgrep.yml --error .  # security rules
```

Tests cover pure Python logic and run without `fzf`, `fd`, `rga`, or SSH.

### 7.2. Adding Tests

Add tests to `tests/test_fzfr.py`. The file imports from both the `src/fzfr/`
package (development) and the built `fzfr` script (CI). Both paths must work.

When a test deliberately triggers a warning or error, suppress stderr to keep
`make build` output clean:

```python
with contextlib.redirect_stderr(io.StringIO()):
    result = _parse_extensions("py;evil")
self.assertEqual(result, [])
```

## 8. Commit Message Style

Commit messages follow the Conventional Commits specification.

Format: `type(scope): subject`

| Type | Use for |
|------|---------|
| `feat` | New user-visible feature |
| `fix` | Bug fix |
| `refactor` | Code change with no behaviour change |
| `docs` | Documentation only |
| `style` | Formatting, whitespace, comment style |
| `test` | Adding or updating tests |
| `build` | Build script or dependency changes |
| `ci` | CI workflow changes |
| `chore` | Housekeeping (gitignore, rename, delete) |
| `security` | Security fix or hardening |

Scope is optional but encouraged: the module or area affected
(`remote`, `config`, `build`, `actions`, `tests`).

Subject: lowercase, imperative, no trailing period. 72 chars max.

Body: explain *why*, not *what*. Reference the rule or constraint that
motivated the change (e.g. "semgrep fzfr-subprocess-shell-true",
"Ruff E741", "WORK_BASE symlink check").

Examples:
```
feat(actions): add overlay output mode for custom actions
fix(build): use ast.walk for import collection to avoid docstring false positives
refactor(config): extract _validate_group and _validate_action helpers
security(remote): assert SCRIPT_HASH is hex-only before interpolation
chore: gitignore built fzfr script
```

## 9. Common Mistakes and How to Avoid Them

**Build fails with "invalid syntax" at an early line number.**
Almost always caused by an import inside a function body that the build
script strips, leaving an undefined name. Move all imports to module top
level. Check especially `from typing import ...`.

**Build fails with a prose sentence in the stdlib imports block.**
The build script collects imports via `ast.walk` and cannot be fooled by
docstring content. If this happens, a module has a syntax error that makes
`ast.parse` fail, falling back to the old line-scanner. Fix the syntax error.

**semgrep reports `fzfr-subprocess-shell-true`.**
`shell=True` is only acceptable for the user-defined custom action command
string in `cmd_internal_exec`. Add `# nosemgrep: fzfr-subprocess-shell-true`
with a comment explaining why it is safe. Do not add `shell=True` elsewhere.

**Pyright reports "X is not accessed" for an exported function.**
False positive. Functions like `_list_archive`, `_ssh_opts_str`, `_mutate_state`
are used by other modules. Pyright analyses files in isolation. No fix needed.

**fzf redraws/clears the screen after a custom action.**
Use `execute-silent` not `execute` for the leader bind. `execute` triggers a
fzf redraw when the subprocess exits. `execute-silent` does not.

**Path works locally but fails remotely.**
Paths from fzf are relative to `base_path`. Use `_abs_path()` in `internal.py`
to resolve them: locally via `Path`, remotely via `PurePosixPath` (no local
filesystem access).

**Tests are noisy during `make build`.**
Tests that deliberately trigger warnings must suppress stderr with
`contextlib.redirect_stderr(io.StringIO())`. See section 7.2.

## 10. Handling Ambiguity

These instructions cannot cover every situation. When you encounter ambiguity:

1. **Stop and state the ambiguity clearly.** Describe the problem and the
   conflicting options you see.
2. **Propose a solution.** Based on the project principles (wiring layer,
   graceful degradation, single-file architecture), recommend one option and
   explain your reasoning.
3. **Await a directive.** Do not implement until you receive explicit
   instructions from the Architect.

Examples of situations that require stopping:
- Adding a new mandatory dependency.
- Changing the build process or module concatenation order.
- A security fix that requires breaking the public config schema.
- A feature that cannot be expressed as a custom action but is borderline
  for core inclusion.

Do not interpret silence or vague approval as a directive to proceed with
significant architectural changes.
